# 电商生图工作台设计规格

## 目标

复刻参考电商生图网站的核心功能，但重做 UI 和信息结构。第一版做成专业卖家工作台：用户进入后可以上传商品图，选择生成用途，配置平台和尺寸，创建生成任务，查看结果、失败原因、下载记录和历史任务。

第一版采用半真实实现：上传、任务状态、历史记录、导出控制是真实流程；AI 生图先通过可替换的 mock provider 返回示例结果，后续接真实模型接口时保持页面和任务数据结构不变。

## 产品边界

P0 完整实现：

- 商品主图生成入口：白底图、平台尺寸图、生活方式场景图。
- 详情页长图生成入口：卖点、规格、场景、对比模块组合。
- 任务与历史记录：任务状态、失败原因、重试、下载记录、消耗记录。
- 工作台参数：平台、尺寸、风格、输出格式、商品卖点、规格信息。

P0 简化实现：

- 商品视频入口：展示上传、脚本、分镜、预览的流程，但不做真实视频生成。
- 在线编辑器：第一版只做结果预览、尺寸/格式导出控制，不做复杂画布编辑。

暂不做：

- 复杂视频生成。
- 团队协作和审核流。
- ERP、店铺后台、商品库深度集成。
- 完整店铺建站。

## 信息架构

第一屏是工作台，不做营销首页优先。

主导航：

- 工作台
- 模板库
- 历史任务
- 价格
- 账户

工作台布局：

- 左侧：生成模块导航，包括商品主图、白底图、生活方式场景图、详情页长图、Shopify Banner、商品视频入口。
- 中间：商品图上传、示例商品、原图预览、生成结果预览、主要操作按钮。
- 右侧：参数面板，包括平台、尺寸、风格、文案、输出格式。
- 底部：最近任务、失败原因与重试、下载历史与消耗记录。

## 核心页面

### 工作台

工作台是主入口。用户可拖拽或选择商品图，选择生成模块，填写卖点和规格，设置平台和尺寸，然后创建任务。

工作台需要支持：

- 上传商品图并显示预览。
- 使用示例商品快速体验。
- 切换生成模块时保留当前商品图。
- 创建任务后进入处理中状态。
- 任务完成后展示 mock 结果图和导出操作。
- 任务失败时展示明确原因和重试按钮。

### 模板库

模板库用于展示可选生成方向，不作为第一版的主流程入口。模板卡片按用途分类：Amazon 主图、Shopify Banner、详情页、生活方式场景、功能卖点图。

点击模板后回到工作台，并预填生成模块、尺寸、风格和提示词结构。

### 历史任务

历史任务用于卖家复用之前的商品图和参数。列表显示任务类型、状态、创建时间、消耗、结果缩略图和失败原因。

支持：

- 按任务状态筛选。
- 重新生成。
- 下载结果。
- 复用参数回到工作台。

### 价格与账户

第一版可以先做静态价格页和基础账户入口。任务消耗记录在本地或 mock API 中保存，为后续真实支付和配额系统留接口。

## 数据模型

### ProductInput

- id
- imageUrl
- fileName
- createdAt
- source: upload 或 sample

### GenerationConfig

- module: main_image, white_background, lifestyle, detail_page, shopify_banner, video_preview
- platform: amazon, shopify, independent_store
- aspectRatio: 1:1, 4:5, 16:9, long_page
- style: studio, lifestyle, premium, minimal
- outputFormat: png, jpg, webp
- sellingPoints
- specifications

### GenerationTask

- id
- productInputId
- config
- status: idle, queued, processing, completed, failed
- resultUrls
- errorCode
- errorMessage
- creditCost
- createdAt
- completedAt

## 数据流

1. 用户上传图片或选择示例商品。
2. 前端创建 ProductInput，并显示预览。
3. 用户选择生成模块和参数。
4. 前端提交 GenerationConfig 创建 GenerationTask。
5. mock provider 根据模块返回对应类型的示例结果。
6. 任务状态从 queued 到 processing，再到 completed 或 failed。
7. 完成后结果进入预览区，并写入历史任务。
8. 用户可下载、重试、复用参数或切换模块继续生成。

## Provider 边界

AI 生成能力必须通过 Provider 接口隔离：

- MockGenerationProvider：第一版使用，返回稳定示例结果和可控失败场景。
- RealGenerationProvider：后续接真实模型接口时实现同一接口。

Provider 不应该直接控制 UI。UI 只依赖任务状态、结果 URL 和错误信息。

## 错误处理

必须展示用户能理解的错误原因：

- 图片过大或格式不支持。
- 图片上传失败。
- 任务创建失败。
- 生成超时。
- 生成失败但未消耗额度。
- 结果保存失败。

失败任务保留在历史记录中，支持重试。若任务未成功完成，不应计入成功生成消耗。

## UI 风格

整体风格应偏跨境卖家工具，不做国内促销海报感。

设计原则：

- 信息密度高但清晰。
- 主色保持克制，避免大面积单一紫蓝渐变。
- 第一屏突出商品图、参数和结果，不放大幅营销 hero。
- 卡片只用于模板、任务、结果缩略图，不做嵌套卡片。
- 主要操作按钮明确区分生成、导出、重试。
- 移动端使用纵向工作流：上传、参数、结果、历史依次堆叠。

## 验证标准

实现完成后至少验证：

- 能启动本地站点。
- 上传图片后能预览。
- 示例商品能加载。
- 主图和详情页模块能创建任务并返回 mock 结果。
- 失败任务能展示原因并重试。
- 历史任务能记录完成和失败任务。
- 导出控制不会导致页面布局错乱。
- 桌面和移动视口不出现文本重叠或按钮溢出。
